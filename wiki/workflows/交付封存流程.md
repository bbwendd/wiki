# 交付封存流程

## 目的

確保每次交付後都能追溯來源、版本、評估結果與後續待辦，避免交付物散落在對話、附件或臨時檔案中。

## 輸入

- Confirmed Plan。
- 最終交付物。
- 來源清單：raw、wiki、schema、外部連結或 connector 結果。
- ABCDE 評估。
- 使用者確認或修正意見。

## 處理步驟

1. 確認 Evaluator Gate 已通過。
2. 將交付物連回對應專案頁、人物頁或決策頁。
3. 在專案頁記錄交付日期、版本、用途與狀態。
4. 保留來源路徑與外部來源日期。
5. 若交付物產生新待辦，使用標準 checkbox 格式寫入專案頁或工作總覽。
6. 若使用者修正格式或語氣，判斷是否需要沉澱到 `schema/templates` 或 `schema/*_RULES.md`。
7. 若流程可重複，更新 `wiki/workflows` 或 `automations/AUTOMATION_STATUS.md`。

## 輸出

```md
## 交付封存紀錄

- 交付物：
- 專案：
- 日期：
- 版本：
- 狀態：已交付 / 待確認 / 需修正
- 來源：
- ABCDE 結論：
- 後續待辦：
- 是否需要更新 schema / workflow / automation：
```

## 需要確認的地方

- 交付物是否已被使用者接受。
- 是否需要產出可下載檔案或外部系統版本。
- 是否涉及對外寄送、發布、分享或承諾。
- 是否有新增機密或敏感資料需要限制擴散。

## 常見失敗情境

- 只在對話中交付，沒有連回專案頁。
- 沒有保存 ABCDE 評估，後續無法知道為何可交付。
- 交付後產生新待辦，但未寫入追蹤表。
- 使用者修正了格式，但未沉澱成模板。
- 外部來源沒有日期或版本，日後難以查證。
