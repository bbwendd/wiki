# 分層規則

本知識庫固定分為 `raw / wiki / schema / automations` 四個操作層。前三層是核心知識架構，`automations` 是由穩定工作流延伸出的執行層。

## 1. raw：原始證據層

### 目的

保存未被改寫的來源資料，讓所有整理、判斷、草稿與決策都能回查來源。

### 放入條件

- Email 原文、寄件備份、重要回覆。
- 會議逐字稿、會議紀錄、錄音摘要。
- 專案附件、需求文件、簡報、計畫書。
- 使用者口述的背景、偏好、決策。
- 外部網站、GitHub、Drive、Slack、Calendar 等來源摘要。

### 禁止事項

- 不在 raw 裡做漂亮總結取代原文。
- 不刪除原始內容。
- 不把多個版本合併成無法追溯的單一版本。

### 必要欄位

- 日期
- 來源
- 相關專案
- 相關人物
- 狀態
- 原始內容或來源描述
- 可抽取事項：事實、待辦、風險、待確認

### 完成標準

- 未來能知道這筆資料從哪裡來。
- 能支持 wiki 頁面的事實與決策。
- 缺資料已標成 `待確認`。

## 2. wiki：可用知識層

### 目的

把 raw 的資料蒸餾成日常可用的工作記憶，用於查詢、決策、寫作、會議準備、專案管理與交接。

### 放入條件

- 未來會再次查找。
- 會影響專案進度、期限、風險、待辦或對外溝通。
- 涉及重要人物、組織、合作關係或決策。
- 可作為文件、簡報、Email、會議紀錄的背景。

### 子層規則

- `wiki/projects`：放專案狀態、背景、目標、待辦、風險、產出文件、決策、來源索引。
- `wiki/people`：放人物、單位、角色、偏好、互動紀錄、待追蹤。
- `wiki/decisions`：放已確認決策、日期、來源、影響與後續待辦。
- `wiki/workflows`：放可重複執行的流程。
- `wiki/dashboards`：放跨專案掃描總覽，不放長篇證據。

### 禁止事項

- 不把待確認內容寫成已確認事實。
- 不用 dashboard 取代專案頁。
- 不用 wiki 取代 raw 的證據保存。

### 完成標準

- 每個重要頁面有 metadata。
- 每個專案能看出下一步、期限、風險與來源。
- 任務使用統一 checkbox 格式。
- 重要事實能追到 raw 或外部來源。

## 3. schema：規則與模板層

### 目的

保存 AI 要遵守的規則、模板、輸出格式、檢查表、工作方法與使用者偏好。

### 放入條件

- 使用者反覆要求的格式。
- 已確認好用的輸出範本。
- 專案頁、人物頁、每日報告、任務格式等模板。
- AI 容易犯錯、需要固定檢查的規則。
- 新的工作方法、判斷標準、命名規則。

### 禁止事項

- 不把單次專案細節放入 schema，除非它是可重複規則。
- 不把未驗證偏好寫成永久規則。
- 不讓 schema 變成長篇知識庫；具體內容應放 wiki 或 raw。

### 完成標準

- 下一個 AI 進來能照規則工作。
- 規則能被引用到具體模板或 workflow。
- 使用者修正過的習慣已沉澱成可重複規則。

## 4. automations：自動化執行層

### 目的

登記可排程或可事件觸發的工作流程。

### 放入條件

- 每天、每週或固定事件後都會重複做。
- 有清楚輸入來源、輸出格式與失敗處理。
- 沒有命中時也需要回報。
- 對外行動可以維持先草稿後確認。

### 禁止事項

- 不自動寄 Email。
- 不自動傳訊息。
- 不自動建立、修改、取消會議。
- 不自動刪除或移動重要檔案。
- 不自動對外承諾交期、價格、範圍或合作條件。

### 完成標準

- 目的、頻率、輸入、輸出、確認規則、失敗回報都明確。
- 有 no-hit 回報文字。
- 可追溯到對應 workflow 或 schema。
